## CORE PHILOSOPHY & COMPETITIVE POSITIONING

WebCore is not "just another SEO tool" or "just another security scanner." WebCore is the **unified diagnostic operating system** for any web property. It answers every question a site owner, developer, or security engineer could ask about a URL — and surfaces insights they didn't know to ask.

**Competitive landscape (2026):**
- **CheckVibe** — security (100+ checks), SEO (68), AEO (46), uptime, Core Web Vitals, accessibility, email monitoring. One scan ~30s. Strongest current all-in-one.
- **Ahrefs / Semrush** — deep SEO but no security, no AEO depth.
- **Pentest-Tools / Burp / Nuclei** — deep security but no SEO/AEO.
- **PerfBee** — pure performance (Lighthouse, CrUX, competitor benchmarks) but no security or AEO.
- **Profound / Surfer SEO** — AEO-focused but no security or performance.

**WebCore's differentiator:** WebCore does **all of the above** in one unified scan **plus** predictive analytics, historical trending, AI-generated remediation, multi-engine AEO matrix, SPA-aware discovery, source-map secret extraction, and a live monitoring dashboard — all powered by Firebase's real-time infrastructure on a lean Vercel deployment. WebCore is the **single pane of glass** for web health.

### MODULE 1: SECURITY SCANNER (150+ checks)

**Objective:** Identify every security weakness in the target website.

**Sub-checks organized by category:**

#### 1.1 HTTP Security Headers
- Strict-Transport-Security (HSTS), Content-Security-Policy (CSP) directive analysis, X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, Referrer-Policy, Permissions-Policy, Cross-Origin-Embedder-Policy (COEP), Cross-Origin-Opener-Policy (COOP), Cross-Origin-Resource-Policy (CORP), CORS misconfiguration detection, Cache-Control analysis, Set-Cookie flags (Secure, HttpOnly, SameSite)

#### 1.2 SSL/TLS Configuration
- Certificate validity, chain completeness, TLS protocol support (1.2, 1.3), cipher strength, HSTS preload readiness, OCSP stapling, Certificate Transparency (SCTs)

#### 1.3 OWASP Top 10 (2021) Checks
- A01-A10 coverage: Broken Access Control, Cryptographic Failures, Injection (SQLi, NoSQLi, Command, SSTI), Insecure Design, Security Misconfiguration, Vulnerable Components (CVE database lookup), Auth Failures, Integrity Failures, Logging/Monitoring gaps, SSRF probing

#### 1.4 Source Map & Secret Leak Detection
- JS source map discovery, parsing for hardcoded secrets (API keys for Stripe, OpenAI, Firebase, AWS, Google Cloud, GitHub tokens, Discord webhooks, etc.), JWT tokens, DB connection strings, env pattern leaks — dev vs production classification

#### 1.5 BaaS & Third-Party Misconfigurations
- Supabase exposed keys, Firebase open Firestore rules, Auth0 misconfigurations, Stripe key leakage, OpenAI key exposure

#### 1.6 Dependency & CVE Scanner
- JS library fingerprinting → NVD/CVE cross-reference, CVSS scoring, KEV catalog prioritization, end-of-life library flagging

#### 1.7 General Web Security
- CSRF detection, open redirect, subdomain takeover, email spoofing (SPF/DKIM/DMARC), mixed content, form action analysis, iFrame sandboxing, server info disclosure

#### 1.8 API Security
- API endpoint discovery, auth requirement checks, rate limiting detection, CORS analysis, GraphQL introspection detection

## SCAN PIPELINE EXECUTION FLOW (Vercel + Firebase)

```
1. User submits URL → Next.js API Route: POST /api/v1/scan
2. URL validation (normalize, resolve, check reachability)
3. Create Scan document in Firestore /scans/{scanId} (status: "queued")
4. Return scanId to frontend immediately
5. Frontend starts Firestore onSnapshot listener on /scans/{scanId}

6. Next.js API Route triggers background scan via one of:
   ├── Vercel background function (res.send() early, continue processing)
   ├── Upstash QStash (HTTP-based queue with retries)
   └── Or synchronous in a single API route (with 60s Vercel timeout)

7. Scan execution happens inside Next.js API Routes (serverless functions):

   ├── Module A: Security Scanner (axios + custom checks)
   ├── Module B: SEO + AEO Scanner (Playwright via browserless or fetch-based lightweight checks)
   ├── Module C: Performance Scanner (Lighthouse via chrome-launcher or CrUX API only for lightweight)
   ├── Module D: Infrastructure + Indexing (dns module, SPF/DKIM parsers, fetch)
   ├── Module E: AI Readiness + Competitor (Genkit flows via API route)
   └── Each module writes results to Firestore subcollections and updates progress

8. Genkit flows run inside Next.js API routes (Gemini API calls)
9. After all modules complete → aggregate scores, compute WebCore Index
10. Update /scans/{scanId} with final scores, status: "complete"
11. Firestore onSnapshot triggers frontend update automatically
12. Post-completion: webhooks (fetch to Slack/Discord), email (Resend/SendGrid), GitHub issues
13. If monitoring enabled → create /monitors/{scanId}
```

## FIRESTORE SECURITY RULES

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
      allow read: if request.auth != null && 
        get(/databases/$(database)/documents/organizations/$(resource.data.organizationId)).data.members.hasAny([request.auth.uid]);
    }
    match /scans/{scanId} {
      allow read: if request.auth != null && (
        resource.data.userId == request.auth.uid ||
        get(/databases/$(database)/documents/organizations/$(resource.data.organizationId)).data.members.hasAny([request.auth.uid])
      );
      allow create: if request.auth != null && request.resource.data.userId == request.auth.uid;
      allow update: if request.auth != null && resource.data.userId == request.auth.uid;
      allow delete: if request.auth != null && resource.data.userId == request.auth.uid;
      match /{document=**} { allow read, write: if request.auth != null; }
    }
    match /monitors/{scanId} {
      allow read, write: if request.auth != null;
    }
    match /organizations/{orgId} {
      allow read: if request.auth != null && resource.data.members.hasAny([request.auth.uid]);
      allow write: if request.auth != null && resource.data.ownerId == request.auth.uid;
      allow create: if request.auth != null;
    }
    match /billing/plans/{planId} { allow read: if true; }
  }
}
```

## IMPLEMENTATION NOTES

1. **Read `design.md` FIRST** — defines all UI.
2. **Vercel serverless timeout** (60s default, 300s with Pro) — for long scans, break into multiple API calls or use background functions.
3. **Playwright limitation** — Vercel serverless doesn't support headless Chromium well. Use **browserless.io** (SaaS) or a lightweight fetch-based analysis instead of full browser renders for initial version.
4. **Firebase Admin SDK** — initialize in a shared utility (`lib/firebase-admin.ts`). Store credentials in Vercel env vars.
5. **Real-time progress** — client uses `onSnapshot` on the scan document. Each module API route writes progress updates.
6. **Genkit** — runs as direct API calls within API routes. Use `genkit` package with Gemini.
7. **Cron jobs** — set up in `vercel.json` for scheduled rescans and uptime checks.
8. **Firestore indexes** — add composite indexes for scans by `userId + createdAt` and `organizationId + createdAt`.
9. **Rate limiting** — implement via Vercel middleware or a simple counter in Firestore.
10. **API key auth** — verify custom tokens via Firebase Auth Admin SDK in API routes.
